# GUI Plan

## 背景

当前项目已经具备命令行形态下的核心链路：

1. 截图屏幕
2. 手动选择弹幕区域
3. 识别弹幕
4. 生成回复
5. TTS 播放

但当前仍然是开发者工作流，依赖：

- `pyenv-win`
- `.venv`
- `pip install`
- `.env`
- `python -m src.main`

这不适合最终分发给普通用户。

## GUI 的产品目标

后续应把项目从“命令行原型”推进到“普通用户可使用的桌面应用”。

GUI 的目标不是单纯加个界面，而是解决以下问题：

- 用户不需要理解 Python 环境管理
- 用户不需要手动编辑一堆配置文件才能运行
- 用户可以通过界面完成启动、停止、选区、查看状态
- 用户可以直接修改 `SOUL.md`、`IDENTITY.md`、`USER.md`
- 用户可以查看当前监控区域和最近识别结果
- 后续便于打包成 Windows 安装包

## 技术路线建议

推荐使用 `PySide6` 作为 GUI 框架。

原因：

- Python 原生生态，能直接复用当前项目代码
- 适合桌面应用形态
- 后续打包分发相对成熟
- 可以兼容 Windows 和 macOS

不推荐把 `tkinter` 作为长期 GUI 主方案，因为当前项目已经踩到 Tk/Tcl 运行时稳定性问题。

## 第一版 GUI 要解决的核心问题

第一版 GUI 不追求复杂，而是优先覆盖最基础的桌面可用性：

1. 启动 / 停止程序
2. 选择监控区域
3. 查看当前运行状态
4. 编辑人格和提示词文件
5. 编辑常用配置
6. 查看日志和最近结果

## 第一版界面信息架构

建议先做一个主窗口，包含以下 5 个主要区域或页面。

### 1. 控制台

功能：

- 启动按钮
- 停止按钮
- 当前状态显示：未启动 / 运行中 / 等待选区 / 异常
- 当前使用的模型信息
- 当前 TTS 提供商
- 当前区域坐标
- 最近一条错误信息

目标：

- 让用户能像使用普通桌面软件一样控制程序，不需要终端命令

### 2. 监控区域

功能：

- “选择区域”按钮
- 当前区域坐标显示
- 整体截图预览
- 裁剪后的弹幕区域预览
- “重新选择区域”按钮

目标：

- 让用户能看到程序到底在监控哪里
- 让用户在界面里完成区域重选，而不是依赖命令行

### 3. 人格与提示词

功能：

- 编辑 `agent_config/SOUL.md`
- 编辑 `agent_config/IDENTITY.md`
- 编辑 `agent_config/USER.md`
- 保存按钮
- 未保存提醒

后续可扩展：

- 系统提示词
- 回复风格参数
- 屏蔽词 / 黑名单词

目标：

- 让非技术用户也能修改角色设定，不需要自己进目录找 markdown 文件

### 4. 运行配置

功能：

- 视觉模型配置
- LLM 配置
- TTS 配置
- 截图频率
- 超时时间
- TTS 采样率
- 测试 TTS 按钮
- 保存配置按钮

目标：

- 把常用配置从 `.env` 里搬到可视化界面

### 5. 运行日志

功能：

- 实时日志输出
- 最近识别的弹幕
- 最近生成的回复
- 最近音频文件路径
- 可选展示 `runtime/memory/session_memory.json` 的摘要

目标：

- 让用户和开发者都能快速判断当前程序是不是正常工作

## 第一版 GUI 的最小功能范围

建议第一版只做以下功能：

- 主窗口
- 启动 / 停止
- 手动区域选择
- `SOUL.md` / `IDENTITY.md` / `USER.md` 编辑
- 常用配置编辑
- 日志查看
- 截图 / 裁剪预览

先不要做：

- 多账户
- 云同步
- 历史任务管理
- 复杂插件系统
- 过重的页面导航结构

## 代码结构建议

不要把 GUI 逻辑直接塞进现有业务模块，建议拆成 4 层。

### 1. core

负责当前已有业务逻辑：

- 截图
- 识别
- LLM
- TTS
- 主循环
- 记忆管理

### 2. services

对 GUI 暴露稳定接口，例如：

- `start_runtime()`
- `stop_runtime()`
- `select_region()`
- `load_settings()`
- `save_settings()`
- `load_prompt_files()`
- `save_prompt_files()`

### 3. gui

负责：

- 窗口
- 页面
- 表单
- 日志展示
- 图片预览

### 4. platform

负责平台相关能力：

- Windows / macOS 截图差异
- 区域选择实现
- 音频播放
- 快捷键

## 运行模型建议

GUI 不应直接阻塞在主线程里跑现有 runtime。

建议路线：

- GUI 主线程负责界面
- 后台线程或子进程运行主循环
- 通过事件、信号或消息机制把状态和日志回传 GUI

两种方式里，更推荐最终走“GUI + 后台子进程”：

- 更稳定
- runtime 崩了不会直接拖死 GUI
- 更接近后续安装包形态

## 配置管理建议

当前很多配置在 `.env`。

如果未来面向普通用户，建议逐步调整为：

- 用户常用配置写到 `runtime/settings.json`
- 开发者兼容配置继续保留 `.env`
- GUI 默认读写 `settings.json`
- 启动时统一映射成现有的 `AppConfig`

`SOUL.md`、`IDENTITY.md`、`USER.md` 可以继续保留文件形式，GUI 只负责编辑它们。

## 跨平台策略

`PySide6` 本身可以兼容 Windows 和 macOS。

但项目里的系统能力不是天然无差异的，尤其是：

- 截图
- 区域选择
- 音频播放
- 快捷键

因此建议策略是：

1. GUI 架构按跨平台设计
2. 第一版先保证 Windows 可用
3. 后续再逐步补 macOS 的底层适配

这意味着：

- GUI 不需要为 macOS 重写一套
- 但底层平台能力需要做条件分支和适配

## 分发路线建议

建议分三阶段推进。

### 阶段 1：命令行 MVP

当前阶段，重点是验证核心链路能跑。

### 阶段 2：GUI MVP

引入 `PySide6`，把命令行工作流包成桌面界面，先给内部测试或少量用户试用。

### 阶段 3：正式分发包

后续做：

- Python 程序打包成可执行文件
- 外部依赖随包分发或首次启动自动准备
- 安装包制作
- 普通用户双击安装、双击启动

## 和今天问题的关系

今天已经确认：

- Tk/Tcl 在当前 Windows Python 环境里不稳定
- Windows 默认播放器不适合严格串行 TTS

因此后续 GUI 和分发方案都要尽量避免依赖“用户本机 Python 环境是否完好”。

这也是为什么 GUI 长期方案不建议继续押在 `tkinter` 上，而更建议使用 `PySide6`，并在分发阶段把运行时和依赖一起带走。
