# Session Summary Template

这个文件用于约定本项目每次开发结束后的总结方式。

以后如果你说：

- “按 `SESSION_SUMMARY_TEMPLATE.md` 更新总结”
- “按固定交接模板更新 README / handover”
- “把今天的问题、方案、TODO 一起补齐”

就按本文件的结构执行，而不是只写一句笼统 TODO。

## 目标

总结文档必须满足两个要求：

1. 当前对话结束后，隔一天再看也能迅速恢复上下文。
2. 即使后续换人接手，也能看懂今天到底遇到了什么问题、为什么这样改、后面还要继续做什么。

## 更新范围

每次需要做“阶段总结”时，至少同步更新以下内容：

1. `handover.md`
2. `README.md` 中与当前结论直接相关的部分
3. 如有必要，补充运行说明、注意事项或 TODO

不要只补其中一个文件，除非用户明确要求。

## 固定输出结构

每次总结至少覆盖以下几部分。

### 1. 今天实际遇到的问题

必须写清楚：

- 具体报错或异常现象
- 触发条件
- 为什么它是问题
- 影响到什么核心能力

不要写成泛泛而谈的“有兼容性问题”。
要写成可定位的描述，例如：

- Windows 下 `tkinter` 可导入，但 `tk.Tk()` 因 `Tcl/Tk` 初始化失败直接报错
- `os.startfile()` 播放音频是非阻塞的，因此无法保证上一条语音播放完再进入下一轮

### 2. 原因判断与结论

必须明确写出这次排查后的判断结果，包括：

- 问题是在代码逻辑、运行环境、第三方依赖、系统权限，还是分发方式
- 哪些猜测已经被排除
- 最终确认的根因或当前最可信结论

如果不是 100% 确认，也要写“当前判断”而不是装作确定。

### 3. 这次已经落地的修复

必须写清楚：

- 改了哪些文件
- 每个改动解决什么问题
- 当前方案是长期方案还是 MVP 临时方案

推荐格式：

- 文件路径
- 改动内容
- 作用

### 4. 已验证结果

必须写清楚这次实际验证过什么，而不是只写“已修复”。

例如：

- 是否成功启动
- 是否成功进入主循环
- 是否真实触发过区域选择
- 是否真实播放过两段音频并确认阻塞顺序
- 是否跑过 `compileall` 或其他基本验证

### 5. 当前仍存在的问题 / 边界

必须单列写出还没解决的问题，避免后续接手的人误以为已经彻底完成。

例如：

- 只是开发态可用，还不适合分发
- 仍依赖某个系统工具
- 仍未做窗口级捕获
- 仍无自动化测试

### 6. 最终想实现的产品目标

如果这次讨论已经涉及“长期方案”或“分发目标”，必须写明最终目标，而不只是写眼前补丁。

例如：

- 最终交付物应是 Windows 安装包或桌面应用
- 用户应开箱即用，而不是面对 `.venv`、`pip install`、`ffmpeg`

### 7. 后续 TODO

TODO 不能只写一句话，必须拆成可执行事项。

每条 TODO 要尽量回答：

- 要做什么
- 为什么要做
- 做完后解决什么问题

如果 TODO 涉及多步流程，要按阶段拆开，例如：

1. 自动检测 `ffplay`
2. 缺失时自动下载并缓存
3. 播放逻辑优先使用项目内缓存
4. 后续做成安装包

## 写法要求

- 少写空话，不写“优化体验”“提升稳定性”这种没有上下文的句子。
- 多写具体事实、具体约束、具体文件、具体行为。
- 如果这次只是 MVP 修复，要明确标注“这是 MVP 方案，不是最终分发方案”。
- 如果用户已经表达了产品目标，例如“最终要做成给普通用户安装的分发包”，必须把这个目标写进总结。

## 默认执行规则

以后只要用户提出以下任一意图，就默认按本模板执行：

- 要求“总结一下今天做了什么”
- 要求“更新交接文档”
- 要求“补一下 TODO”
- 要求“把问题和后续方案写下来”
- 要求“方便以后换人接手”

## 推荐口令

以后你可以直接说：

- “按 `SESSION_SUMMARY_TEMPLATE.md` 更新今天的交接”
- “按固定总结模板补 handover 和 README”
- “把今天的问题、修复、验证、TODO 按模板写进去”

看到这些说法时，就按本文件执行。
